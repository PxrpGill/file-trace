# Загрузка директорий и распаковка zip/rar-архивов

Дата: 2026-07-06

## Проблема

Сейчас `UploadFileButton` умеет грузить только плоский список отдельных файлов
(`POST /api/folders/{folder_id}/files`, один файл — один запрос). Пользователю
негде разом залить целую директорию с подпапками, а zip/rar-архив можно только
хранить как обычный файл — развернуть его содержимое в дерево папок нельзя.

## Решение

### 1. Загрузка директории

#### Backend: `POST /api/folders/{folder_id}/upload-tree`

Новый эндпоинт в `app/api/files.py`, принимает `files: list[UploadFile]`. Имя
каждого `UploadFile` — относительный путь с `/` (пример: `docs/report.pdf`);
именно так браузер передаёт `webkitRelativePath`, если явно указать его как
`filename` при добавлении в `FormData` (см. фронтенд-часть).

Права проверяются один раз в начале — `require_folder_access(db, user,
folder_id, PermissionLevel.write)` (право на папку действует на всё
поддерево, повторных проверок на вложенных уровнях не требуется).

Для каждого файла из списка, последовательно:

1. Разбить путь на сегменты по `/`; отбросить пустые сегменты и сегменты `.`
   / `..` (защита от «выхода» из целевой папки — это чисто логическая
   иерархия папок в БД, а не файловая система, но сегмент `..` не должен
   давать пользователю возможность как-то обмануть структуру).
2. Пройти по сегментам-папкам от `folder_id`: на каждом уровне найти дочернюю
   `Folder` с таким именем; если нет — создать (`audit.record(...,
   AuditAction.folder_create, ...)`, тот же паттерн, что в
   `app/api/folders.py:create_folder`), если есть — переиспользовать.
3. Последний сегмент — имя файла. В получившейся конечной папке:
   - если файла с таким именем нет — создать `File` + первую `FileVersion`
     (как `upload_file`), записать `AuditAction.file_upload`;
   - если файл с таким именем уже существует — добавить новую
     `FileVersion` (как `upload_new_version`), записать
     `AuditAction.file_new_version`. (Конфликт имён при массовой загрузке —
     это норма, не ошибка 409, в отличие от одиночной загрузки одного
     файла.)

Общий helper для «создать-или-переиспользовать вложенную папку по списку
сегментов» переиспользуется и в распаковке архивов (часть 2) — выносится в
`app/services/tree_upload.py` (или аналогичное новое имя) вместе с общей
функцией сохранения файла с логикой upload/new-version.

Все изменения — в одной транзакции, один `db.commit()` в конце запроса. Если
при обработке любого файла возникает ошибка (например, повреждённый поток
данных) — весь запрос откатывается и возвращает 400; частичный успех в первой
версии не поддерживается (проще и предсказуемее).

#### Frontend

Новая кнопка «Загрузить папку» рядом с существующей «Загрузить файл»
(`frontend/src/features/file/upload-file/` — новый компонент в соседней
директории `upload-tree/`, по аналогии). Использует скрытый `<input
type="file" webkitdirectory multiple>`.

`onChange`: собрать `FormData`, для каждого выбранного файла —
`form.append('files', file, file.webkitRelativePath)` (третий аргумент
`append` явно задаёт имя во внутренней части multipart-запроса, включая `/` —
так путь долетает до бэкенда неповреждённым). Отправить один запрос `POST
/api/folders/{folderId}/upload-tree`.

Прогресс — один общий индикатор на кнопке, использующий тот же паттерн, что
`UploadVersionButton` (прогресс-бар вместо кнопки, `onUploadProgress` по
общему объёму данных запроса). После успеха — инвалидация `['tree']` и
`['files']`.

### 2. Архивы: хранение и распаковка

#### Хранение — без изменений

Zip/rar-файл загружается как обычный файл через уже существующие
`upload_file` / `upload_new_version` — эти эндпоинты не меняются.

#### Backend: `POST /api/files/{file_id}/extract`

Новый эндпоинт в `app/api/files.py`.

1. `_get_file(db, user, file_id, PermissionLevel.write)` — доступ на запись к
   папке, где лежит файл.
2. По расширению имени файла (регистронезависимо) выбрать модуль:
   `.zip` → stdlib `zipfile`; `.rar` → `rarfile` (новая зависимость, см.
   ниже). Любое другое расширение → 400 «Формат не поддерживается».
3. Открыть архив из хранилища (`storage.open(version.storage_key)`).
   Прочитать список записей (`ZipInfo`/`RarInfo`) **без распаковки** —
   у обоих есть `file_size` в метаданных.
4. Проверки перед распаковкой (все — без записи на диск):
   - **Zip-slip**: если путь любой записи абсолютный, начинается с `/`, или
     содержит сегмент `..` — прервать, 400 «Архив содержит недопустимые
     пути», ничего не создавать.
   - **Лимиты** (защита от zip-бомб): суммарный `file_size` по всем записям
     > 2 ГБ → 413; количество записей > 10 000 → 413. Ничего не создаётся.
5. Создать (или переиспользовать, если уже существует с таким именем)
   подпапку в папке файла: имя = имя архива без расширения
   (`report.zip` → `report`). Конфликт с уже существующей папкой этого имени
   — не ошибка, распаковываем внутрь неё (переиспользуем `folder_create` per
   существующий паттерн «найти-или-создать», как в п.1).
6. Пройти по записям архива по порядку: директории → создать вложенные
   папки (тем же helper'ом, что в части 1, от корня — новой подпапки);
   файлы → сохранить содержимое через общий helper сохранения файла
   (новый файл → `file_upload`, конфликт имени → `file_new_version`, как в
   части 1).
7. Записать `AuditAction.file_extract` на `file_id` архива с деталями
   `{"name": archive.name, "files": N, "total_size": M}` — фиксирует сам
   факт распаковки отдельно от аудита созданных внутри объектов.
8. Один `db.commit()` в конце. Любая ошибка на любом шаге (проверки,
   повреждённый архив, сбой чтения) — исключение до commit, ничего не
   сохраняется.

Пароль-защищённые архивы, повреждённые файлы — 400 «Не удалось распаковать
архив».

#### rar-зависимость

- `backend/pyproject.toml`: добавить `rarfile` в `dependencies`.
- `backend/Dockerfile`: перед `uv sync` добавить установку системного
  `unar` (`RUN apt-get update && apt-get install -y --no-install-recommends
  unar && rm -rf /var/lib/apt/lists/*`) — `rarfile` использует его как
  бэкенд распаковки.
- Локальная разработка (`make dev`, тесты) не в Docker — README/CLAUDE.md
  дополнить заметкой, что для распаковки `.rar` локально нужен `unar`
  (`brew install unar` / `apt install unar`); тесты на rar можно скипать,
  если бинарник не найден (`pytest.mark.skipif`).

#### Frontend

В `FileDrawer` (или в строке файла в таблице — там, где уже есть другие
file-actions) — новое действие «Распаковать», видимое только когда:
- `canWrite` для текущей папки;
- имя файла оканчивается на `.zip` или `.rar` (без учёта регистра).

Компонент — по аналогии с `DeleteFileAction`/`RenameFileAction`: кнопка →
`POST /api/files/{file.id}/extract` → на время запроса `disabled` для всех
действий над этим файлом (тот же механизм `useIsMutating` с
`mutationKey: ['extract', file.id]`, что уже используется для
`create-version`). После успеха — инвалидация `['tree']` и `['files']`, чтобы
новая подпапка и файлы сразу отобразились.

Ошибки (400/413/409) — показываются через существующий `Modal`
(`onError`), как у остальных file-actions.

### 3. Аудит

Один новый `AuditAction.file_extract` в `app/models/audit.py` + перевод в
`ACTION_LABELS` (`frontend/src/entities/audit/model/action-labels.ts`).
Загрузка директории и создание вложенных папок/файлов при распаковке
полностью укладываются в уже существующие `folder_create` / `file_upload` /
`file_new_version` — отдельных enum-значений для них не нужно.

### 4. Тесты (backend)

- Загрузка директории (`tests/test_files.py` или новый
  `tests/test_upload_tree.py`):
  - вложенные пути создают соответствующие подпапки и файлы, есть
    audit-записи `folder_create`/`file_upload`;
  - конфликт имени файла с уже существующим → новая версия
    (`file_new_version`), не 409;
  - путь с сегментом `..` — сегмент отбрасывается, а не создаёт папку `..`;
  - без write-доступа к `folder_id` → 403.

- Распаковка (новый `tests/test_extract.py`):
  - успешная распаковка zip-фикстуры → верное дерево папок/файлов +
    `file_extract` в аудите;
  - zip-slip (`../evil.txt` внутри архива) → 400, ничего не создано;
  - превышение лимита суммарного размера или числа записей → 413, ничего не
    создано;
  - конфликт имени файла внутри архива с уже существующим файлом в целевой
    папке → новая версия;
  - повторная распаковка в уже существующую одноимённую подпапку →
    переиспользует папку, не падает;
  - `.rar`-фикстура (маленький тестовый архив) — тест скипается, если в
    окружении нет `unar`;
  - без write-доступа к папке файла → 403.

## Сознательно не делаем (YAGNI)

- Рекурсивную распаковку архивов, найденных внутри архива.
- Прогресс распаковки на UI — это разовая серверная операция, клиент просто
  ждёт ответ с заблокированными действиями (как одиночная мутация).
- Частичный успех загрузки директории/распаковки (continue-on-error по
  отдельным файлам) — при ошибке откатывается весь запрос целиком.
- Password-protected архивы — считаются ошибкой распаковки.
- Собственные лимиты на загрузку обычной директории (число/суммарный размер
  файлов) — там нет амплификации сжатия, как в архиве, риска zip-бомбы нет.
