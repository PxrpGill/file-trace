# Sticky-шапка страницы папки в BrowserPage

Дата: 2026-07-08

## Проблема

В `frontend/src/pages/browser/ui/BrowserPage.tsx` правая панель
`<main className="content">` — единственный скролл-контейнер
(`overflow-y: auto` в `styles.css`). Шапка папки (`.content-head`:
название, метка доступа, кнопки «Загрузить файл» / «+ Папка» /
«Переименовать» / «Удалить») и панель массовых действий
(`SelectionToolbar`, класс `.bulk-toolbar`) — обычные блоки в начале
потока. При длинном списке файлов, прокрутив вниз, нужно скроллить
обратно наверх, чтобы нажать любую из этих кнопок.

## Решение

Шапка и панель массовых действий объединяются в один sticky-блок,
прилипающий к верху `.content` при скролле. Баннеры (загрузка файлов,
результат массовой операции) в sticky-зону не входят — они переносятся
ниже sticky-блока, сразу над таблицей файлов.

### 1. Разметка в BrowserPage.tsx

Новая обёртка `<div className="content-sticky">` вокруг существующих
`.content-head` и `<SelectionToolbar>` (только их — без баннеров между
ними). Порядок блоков внутри `.content` меняется на:

1. `.content-sticky` (`.content-head` + `SelectionToolbar`)
2. `upload-banner` (если есть активные загрузки)
3. `bulk-result-banner` (если есть `resultMessage`)
4. `FileTable`

Сейчас баннеры физически находятся между `.content-head` и
`SelectionToolbar` в JSX — при переносе в новый порядок их условная
логика (`uploadingCount > 0`, `resultMessage`) не меняется, меняется
только место рендера.

`SelectionToolbar` уже возвращает `null` при `count === 0`
(`frontend/src/widgets/file-table/ui/SelectionToolbar.tsx:12`), поэтому
sticky-блок при отсутствии выделения состоит из одной шапки — высота
меняется автоматически, без ручного пересчёта отступов.

### 2. Стили в styles.css

Новый класс `.content-sticky`:

```css
.content-sticky {
  position: sticky;
  top: 0;
  z-index: 1;
  background: var(--paper);
  border-bottom: 1px solid var(--line);
}
```

- `position: sticky; top: 0` — блок прилипает к верхней границе
  scroll-контейнера `.content` (у самого `.content` есть верхний
  padding 20px, но при скролле он уходит вместе с остальным
  содержимым, так что блок прилипнет вплотную к видимому краю без
  зазора — дополнительных отступов/минус-маржинов не требуется).
- `background: var(--paper)` — чтобы строки таблицы, уезжающие под
  шапку при скролле, не просвечивали сквозь текст и кнопки.
- `border-bottom` — постоянная тонкая линия отделяет шапку от
  контента; работает и в прилипшем, и в обычном состоянии, без JS
  и `IntersectionObserver` для подсветки «прилип/не прилип».
- `z-index: 1` — подстраховка порядка отрисовки поверх строк таблицы
  (позиционированные элементы и так рисуются поверх обычного потока в
  той же стэкинг-группе, но явный z-index делает это надёжным).

`.content-head` внутри обёртки не меняется (её текущие стили —
`display: flex; align-items: center; gap: 12px; margin-bottom: 16px;
flex-wrap: wrap`). Отступ `margin-bottom: 16px` у `.content-head`
остаётся — он же формирует отступ до `SelectionToolbar`, когда тот
виден.

### 3. Мобильная раскладка

Медиа-запрос `@media (max-width: 720px)` (styles.css:569) не трогает
`.content` — `overflow-y: auto` и, соответственно, sticky-поведение
работают так же, без отдельной адаптации.

## Проверка

- `make build` (`tsc -b && vite build`) — проверка типов и сборка.
- Ручная проверка в браузере через `make dev`: открыть папку с
  достаточным количеством файлов, чтобы появился скролл; прокрутить
  вниз — шапка с кнопками должна остаться видна и кликабельна; выделить
  файлы — панель массовых действий должна прилипнуть сразу под шапкой;
  запустить загрузку файла и/или массовую операцию — баннер должен
  появиться под sticky-блоком, не внутри него; проверить на узком окне
  (мобильная раскладка).

## Вне объёма

- Подсветка/тень, появляющаяся только в момент, когда шапка «прилипла»
  (визуальный эффект через JS/IntersectionObserver) — обходимся
  постоянной border-bottom.
- Изменения бэкенда и аудита — чисто фронтенд/CSS.
